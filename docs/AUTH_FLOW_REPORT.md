# ChapaWorks — Authentication Flow Report
**For:** Design Agent  
**Purpose:** Complete mapping of how artisans and clients reach sign-up/sign-in with different entry points, URL props, and resulting UI states  
**Date:** 2026-05-23

---

## Overview

ChapaWorks uses a **single sign-up page** (`/sign-up`) and a **single sign-in page** (`/sign-in`) for all user types. The **role** (client or artisan) is communicated via:

1. A `?role=artisan` URL query parameter on the sign-up link
2. A `?invite=TOKEN&role=artisan` URL parameter for invite-based artisan signups
3. The absence of a `?role` parameter defaults to **client**

There is **no role selector on the sign-up page** — the role is inferred entirely from the URL. The page adapts its headline, badge, subheadline, and button copy based on which entry point the user came from.

### Admin Invite → `/sign-up?invite=TOKEN&role=artisan`

| Source | Trigger | href | Notes |
|---|---|---|---|
| Admin Dashboard → Artisans → Invites tab | Admin clicks "Send Invite" and sends email | `/sign-up?invite={token}&role=artisan` | Generated by `invite-artisans.tsx`, sent via email |

**Email contains:** A link like `http://localhost:3000/sign-up?invite=abc123&role=artisan`

**Result on sign-up page:** Both `?invite=TOKEN` and `?role=artisan` present
```
Headline:    "Join as an Artisan"
Subheadline: "Showcase your skills, get discovered by clients across Kenya."
Role badge:  🔨 "Signing up as artisan"
Button copy: "Create Artisan Account"
Invite banner: ✉️ "You've been invited to join as an artisan!" (green banner above Google OAuth)
```

**Additional server-side validation:**  
The invite token is fetched from `GET /api/admin/invites/${token}` — if valid, role is forced to `artisan` even if cookie was wrong (race condition protection).

### Admin Bootstrap → `/sign-up` (any entry point)

Admins are not expected to use the sign-up UI in the normal flow. Admin access is granted by:

| Method | How | Notes |
|---|---|---|
| Email bootstrap | Sign up with `meluseno@gmail.com` | Detected via `ADMIN_BOOTSTRAP_EMAILS` env var in `after-sign-up` route |
| Promote existing user | Admin calls `POST /api/user/promote-admin` | Requires existing admin or `ADMIN_PROMOTION_SECRET` env var |

The `meluseno@gmail.com` email auto-gets ADMIN role on first sign-up regardless of which button/URL was used. No special sign-up page appearance.

## Role Assignment Pipeline (After Sign-Up)

```
User clicks CTA button
        │
        ▼
URL formed with role:
  /sign-up              → role = "client" (default)
  /sign-up?role=artisan → role = "artisan"
  /sign-up?invite=TOKEN → role = "artisan" (forced by invite validation)
        │
        ▼
Sign-Up Page (Suspense-wrapped)
  1. useSearchParams() reads ?role param (inside Suspense = no SSR race)
  2. Sets cookie: chapaworks_signup_role = "artisan" | "client"
  3. If ?invite=TOKEN:
     a. Stores invite token cookie: chapaworks_invite_token = TOKEN
     b. Fetches GET /api/admin/invites/TOKEN
     c. If valid → re-sets role cookie to "artisan" (defense-in-depth)
        │
        ▼
Clerk handles account creation
  - Email + password  OR  Google OAuth
  - Email verification (OTP sent)
        │
        ▼
Clerk redirects to /after-sign-up (NEXT_PUBLIC_CLERK_AFTER_SIGN_UP_URL)
        │
        ▼
after-sign-up/route.ts — Role Resolution (4 sources, priority order):
  Source 1: URL ?role param  (future-proofing, not currently set by Clerk Elements)
  Source 2: Cookie chapaworks_signup_role  ← primary mechanism
  Source 3: Clerk publicMetadata.role  (handles repeat visits)
  Source 4: Invite token validation
     → Reads chapaworks_invite_token cookie
     → Queries DB: prisma.artisanInvite.findUnique({ where: { token } })
     → If PENDING and not expired → FORCES role = "artisan"
     → Marks invite as ACCEPTED
  [Special] Admin Bootstrap:
     → Reads Clerk user email
     → If in ADMIN_BOOTSTRAP_EMAILS → FORCES role = "admin"
     → Overrides ALL other sources
        │
        ▼
Set Clerk publicMetadata: { role: "artisan" | "client" | "admin" }
        │
        ▼
Create DB user:
  role = "CLIENT"   → User { role: CLIENT, status: ACTIVE }
                      + Profile { country: "Kenya" }
  role = "ARTISAN"  → User { role: ARTISAN, status: ACTIVE }
                      + Profile { country: "Kenya", artisanStatus: PENDING, isAvailable: false }
  role = "ADMIN"    → User { role: ADMIN, status: ACTIVE }
                      + Profile { country: "Kenya" }
        │
        ▼
Set verified-role cookie: chapaworks_verified_role = role (max-age 120s)
Redirect to /after-sign-in
```

## Post-Sign-In Routing (after-sign-in/route.ts)

```
/after-sign-in
        │
        ▼
Resolve role (3 sources):
  1. session JWT publicMetadata.role  (fastest, already in token)
  2. Clerk API getUser().publicMetadata.role  (JWT propagation delay)
  3. DB lookup: prisma.user.findUnique({ where: { clerkId } }).role  (fallback)
        │
        ▼
Set verified-role cookie: chapaworks_verified_role = role (max-age 120s)
        │
        ▼
Redirect by role:
  "admin"   → /admin-dashboard
  "artisan" → /artisan-dashboard
  "client"  → /client-dashboard
  unknown   → /sign-in  (error state)
```

## Proxy Middleware Enforcement (proxy.ts)

After sign-in, the Clerk middleware in `proxy.ts` enforces role-based routing on every subsequent request to protected routes:

```
Request to protected route (/client-dashboard/*, /artisan-dashboard/*, /admin-dashboard/*)
        │
        ▼
Read role from:
  1. JWT sessionClaims.publicMetadata.role  (fastest)
  2. chapaworks_verified_role cookie  (bridges JWT refresh delay)
        │
        ▼
Role ≠ correct dashboard? → Redirect to correct dashboard
  artisan visiting /client-dashboard  → redirect to /artisan-dashboard
  client visiting /admin-dashboard    → redirect to /client-dashboard
  admin visiting /artisan-dashboard   → redirect to /admin-dashboard
```

Public routes (`/artisans/*`, `/api/search/artisans/*`) are explicitly excluded from role enforcement.

## Sign-Up Page Adaptation Logic (Code Reference)

```typescript
// app/(auth)/sign-up/[[...sign-up]]/page.tsx

// Read role from URL (inside Suspense boundary — no SSR race condition)
const urlRole = searchParams.get('role') === 'artisan' ? 'artisan' : 'client';
const inviteToken = searchParams.get('invite');

// Adapt UI to role
const isArtisan = urlRole === 'artisan';
const headline = isArtisan ? 'Join as an Artisan' : 'Create Your Account';
const subheadline = isArtisan
  ? 'Showcase your skills, get discovered by clients across Kenya.'
  : 'Find and hire verified artisans near you.';

// Button label
`Create ${isArtisan ? 'Artisan' : ''} Account`

// Role badge
<div className={isArtisan ? 'bg-amber-100 text-amber-700' : 'bg-emerald-100 text-emerald-700'}>
  {isArtisan ? <TesseractLogo size={14} /> : null}
  Signing up as {roleLabel}
</div>

// Invite banner (only if invite token present)
{inviteToken && (
  <div className="bg-emerald-50 border border-emerald-200 ...">
    ✉️ "You've been invited to join as an artisan!"
  </div>
)}

// Role switch link at bottom
{isArtisan ? (
  <Link href="/sign-up">Sign up as a client</Link>
) : (
  <Link href="/sign-up?role=artisan">Sign up as an artisan</Link>
)}
```

| Cookie Name | Set By | Value | Max Age | Purpose |
|---|---|---|---|---|
| `chapaworks_signup_role` | Sign-up page (client-side JS) | `"artisan"` \| `"client"` | 3600s (1hr) | Carries role through Clerk's sign-up flow |

## Design Implications for Redesign

### What must stay role-aware
1. **Every CTA button** must carry the correct href (`/sign-up` vs `/sign-up?role=artisan`) — this is the entire mechanism for role assignment
2. **Sign-up page** must clearly communicate which role the user is signing up as (the badge + headline + button copy all change)
3. **Invite banner** needs to be visually distinct and reassuring for artisans arriving from email

### Client-facing pages — default to `/sign-up` (no role param)
- Homepage hero, "Get Started" buttons, public nav
- Browse page (`/artisans`) CTAs
- Artisan profile page CTAs
- Pricing — client/free tier

### Artisan-facing pages — must use `/sign-up?role=artisan`
- `/for-artisans` — ALL buttons on this page
- Pricing — artisan tiers (Monthly, Annual)
- Admin invite emails — generated automatically

### Pages serving both
- `/pricing` — client card → `/sign-up`, artisan cards → `/sign-up?role=artisan`
- Sign-up page itself — role switch link at bottom allows switching

### Sign-in pages
- Role-agnostic — same URL for everyone
- Post-sign-in routing (by `after-sign-in`) handles the different dashboards
- No role information needed in href, no role props needed on sign-in page
